# FFmpeg SDK

TypeScript SDK wrapping the FFmpeg CLI. Designed for reuse across all projects.

## Project Structure

See `ARCH.md` for full architecture. See `INTERFACE.md` for all public interfaces. See `TESTING.md` for testing rules.

```
src/
├── index.ts          # Public barrel export
├── core/             # execute(), probe(), validate
├── hardware/         # GPU detection, session management, fallback
├── encoding/         # Codecs, presets, encoder config builders
├── operations/       # Builder-pattern operations (extract, transform, audio, etc.)
├── convenience/      # Pipeline, batch, smart transcode, thumbnails, waveform
├── filters/          # Filter graph builder, filter string helpers
├── types/            # All type definitions
├── schemas/          # Zod schemas for ffprobe validation
└── util/             # Timecode, temp files, caching, platform detection
```

## Stack

- **Runtime**: Node.js >= 22
- **Language**: TypeScript (strict, ESM only)
- **Build**: tsup (ESM, declarations, sourcemaps)
- **Test**: vitest (unit tests use `.toArgs()`, integration tests require ffmpeg binary)
- **Lint/Format**: biome
- **Validation**: zod
- **Package**: `@ffmpeg-sdk/core`

## Conventions

### Code Style
- Named exports only (no default exports)
- Use `node:` protocol for built-in modules (`node:child_process`, `node:path`)
- Files use kebab-case (`thumbnail-sheet.ts`, `encoder-config.ts`)
- Zod schemas: lowercase module prefix (`probeStreamSchema`, `encoderConfigSchema`)
- Prefer `as const` and literal types over enums (except `FFmpegErrorCode` which is an enum)
- No `as unknown as` type casts
- `import type` for type-only imports

### TypeScript
- `strict: true`, `noUncheckedIndexedAccess: true`
- `target: ES2024`, `module: ESNext`, `moduleResolution: Bundler`
- `verbatimModuleSyntax: true`, `isolatedModules: true`

### Error Handling
- Builder methods throw for programmer errors (missing required fields)
- `.execute()` throws `FFmpegError` for all runtime failures
- `.tryExecute()` returns `{ success: true, data } | { success: false, error }` for callers who prefer Result types
- `FFmpegError` always includes: `code`, `message`, `command`, `stderr`, `exitCode`
- Hardware fallback is automatic in `'auto'` mode - retry with CPU on hwaccel failure

### Patterns
- All operations use fluent builder pattern with `.execute()` terminal
- Every builder has `.toArgs()` for inspection without execution
- Probe results are cached by `(path, mtime)` - use `noCache: true` to bypass
- Hardware detection is cached for process lifetime
- Temp files are auto-cleaned in pipeline operations

### Testing (see TESTING.md for full rules)
- **Every feature that maps to FFmpeg MUST have an E2E test with the real binary**
- Tier 1 (builder): assert on `.toArgs()` output, no ffmpeg binary needed
- Tier 2 (E2E): run real ffmpeg, probe output, verify measurable properties
- Tier 3 (stress): long-running edge cases, gated by `FFMPEG_STRESS_TESTS=1`
- Test media fixtures in `__tests__/fixtures/` (generated by script, < 5 MB total)
- Never mock `execute()` in E2E tests - that defeats the purpose
- Always probe output files - `existsSync` alone is not sufficient
- A feature is NOT done until its E2E test passes

### What Not To Do
- Do NOT bundle ffmpeg binaries - require system installation
- Do NOT add global state - all config is per-instance via `createFFmpeg()`
- Do NOT use `execSync` - all execution is async
- Do NOT swallow errors - always propagate or log + rethrow
- Do NOT include `Co-Authored-By` in git commits
